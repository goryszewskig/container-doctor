Git Tips & Tricks Every DevOps and Data Engineer Should Know

Most developers use maybe 10% of what Git can do. If you work in DevOps or data engineering, the commands below will save you hours — from recovering "lost" work to shipping a hotfix while your desk is full of half-finished changes. Bookmark this one.

First, Decode the Notation

Git commands look cryptic because of symbols like HEAD~5 or stash@{1}. Here's the cheat sheet:

NotationMeaningExample
HEADPointer to your current commit (tip of the checked-out branch)git show HEAD
HEAD~NN commits before HEAD, following first parentsHEAD~5 = 5 commits back
HEAD^First parent of HEAD (HEAD^2 = second parent of a merge commit)git revert -m 1 <sha>
<sha>A commit's hash ID — full 40 chars or a short prefix like a1b2c3dgit cherry-pick a1b2c3d
origin/mainYour local copy of where the remote's main was at last fetchupdated by git fetch
HEAD@{N}Reflog entry — where HEAD was N operations agoHEAD@{2} = two actions ago
stash@{N}Entry on the stash stack, 0 = most recentgit stash apply stash@{1}
A..BCommits reachable from B but not A (linear range)git log main..feature
A...BChanges on B since the branches divergedgit diff main...HEAD
@{u}The upstream branch your current branch tracksgit diff @{u}..HEAD

Common confusion: ~ walks history back by commits, ^ picks a parent (matters for merges), and @{} walks the reflog — what you actually did locally, including discarded work.

Branching & History

Interactive rebase — clean up before you push

git rebase -i HEAD~3
# mark commits 2 and 3 as "squash" to fold them into the first

Fixup commits — fold small fixes into the right commit automatically

git commit --fixup a1b2c3d        # small fix for commit a1b2c3d
git rebase -i --autosquash main   # git places it automatically

Cherry-pick — port one commit to another branch

git checkout release/1.4
git cherry-pick a1b2c3d           # hotfix commit from master

Reflog — your time machine

Deleted a branch? Botched a rebase? Nothing is truly lost for ~90 days:

git reflog
git reset --hard HEAD@{5}         # jump back to a known-good state

Bisect — find the commit that broke everything

git bisect start
git bisect bad                    # current version is broken
git bisect good v1.2.0            # this tag was fine
# git checks out middle commits; mark each good/bad until the culprit is found
git bisect reset

Visual branch map

git log --oneline --graph --all --since="2 weeks ago" --author=me

Daily Efficiency

Modern replacements for checkout

git switch -c feature/new-thing   # create + switch branch
git restore src/app.py            # discard local changes in file

Worktree — two branches, two folders, zero stashing

git worktree add ../proj-hotfix -b hotfix/urgent
cd ../proj-hotfix && make fix && git push
git worktree remove ../proj-hotfix

Three dots — what did I actually change on this branch?

git fetch origin
git diff origin/main...HEAD --stat

Archaeology — who wrote this, and when did that value change?

git blame -L 40,60 -- docker-compose.yml
git log -S "POOL_SIZE" --oneline  # commits that added/removed this text

Aliases that pay for themselves

git config --global alias.lg "log --oneline --graph --all"
git config --global alias.st "status -sb"

Safer Pushing & Syncing

Force-push without the foot-gun

git rebase main
git push --force-with-lease       # refuses if someone else pushed meanwhile

Linear history by default

git config --global pull.rebase true
git config --global fetch.prune true

Create a merge request straight from push (GitLab)

git push -o merge_request.create -o merge_request.target=main

Partial & Precise Staging

Stage (or discard) only the changes you want — even within a single file:

git add -p container_doctor.py    # y/n/s/e per hunk
git restore -p                    # discard selected hunks
git restore --staged secrets.yaml # unstage that one file you didn't mean to add
git commit -v                     # see the diff in the commit editor

DevOps-Specific Patterns

Sparse checkout for monorepos

git clone --filter=blob:none --sparse https://github.com/org/monorepo
cd monorepo
git sparse-checkout set infra/terraform services/api

Shallow clones in CI — faster pipelines

git clone --depth 1 https://github.com/org/repo

End the CRLF/LF wars (especially on Windows)

Add a .gitattributes file:

* text=auto
*.sh text eol=lf
*.py text eol=lf
*.png binary

Pre-commit hooks — lint, format, and scan for secrets automatically

pip install pre-commit gitleaks
pre-commit install                # runs on every git commit

Signed release tags

git tag -s v1.2.0 -m "release 1.2.0"
git push origin v1.2.0

Conventional Commits — unlock automated changelogs

git commit -m "feat(api): add /history endpoint"
git commit -m "fix(db): prevent restart loop on disk-full"

Data Engineering Angle

Committed a secret or a huge CSV? Purge it from history

pip install git-filter-repo
git filter-repo --path credentials.json --invert-paths
git push --force-with-lease
# then rotate the secret anyway — assume it leaked

Large files belong in LFS (or DVC), not plain Git

git lfs install
git lfs track "*.parquet"
git add .gitattributes data/sample.parquet
git commit -m "chore: track sample dataset with LFS"

Debug why a file is ignored

git check-ignore -v data/raw/export.csv

The Recovery Toolkit (Memorize This)

git reflog                      # find lost state
git reset --hard HEAD@{2}       # jump back to it
git revert <sha>                # undo a pushed commit safely
git clean -ndx                  # PREVIEW untracked junk before deleting
git clean -fdx                  # actually delete it

Undo a bad merge that's already pushed:

git revert -m 1 <merge-commit-sha>
git push

Shipping a Hotfix with a Desk Full of WIP

The classic panic scenario: you're mid-feature, everything is uncommitted, and production is on fire. Stash, fix, restore:

# 1. Save your WIP aside (-u includes untracked files, -m labels the stash)
git stash push -u -m "wip: feature X in progress"

# 2. Working tree is clean — create a hotfix branch
git checkout -b hotfix/critical-issue
# ...make the fix...
git add <fixed-files>
git commit -m "fix: critical issue"
git push -u origin hotfix/critical-issue

# 3. Go back and restore your work
git checkout <your-feature-branch>
git stash pop

If the fix touches different files than your WIP, you don't even need stash — just git add the fixed files and commit. Uncommitted work is never pushed.

Rules of Thumb

  1. Rebase your own branches; merge shared branches (main)
  2. Small, atomic commits — one logical change each
  3. git status + git diff --staged before every commit
  4. Rewrite history only before it's pushed or shared
  5. When lost: git reflog first, panic never

Where to start: master rebase -i, reflog, and --force-with-lease. Those three alone will put you ahead of most engineers.
